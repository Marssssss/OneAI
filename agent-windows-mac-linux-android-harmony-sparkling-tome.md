# OneAI Cross-Platform Agent Framework — Implementation Plan

## Context

OneAI is a cross-platform, high-performance Agent framework designed to run on desktop (Windows/Mac/Linux) and mobile (Android/HarmonyOS/iOS). The project's design document specifies Rust as the implementation language, with UniFFI for generating platform-specific bindings. The framework must support multiple LLM providers, advanced Agent paradigms (Plan → Parallel → ReAct → Reflection with state isolation), a Workflow Compiler, tool management with human approval gates, 3-layer output parsing defense, memory systems, RAG, SKILL system, scheduling, and state persistence. Currently the project directory contains only the design document (`OneAI设计.rtf`) — no code exists yet.

## Language Choice: Rust

Rust is the correct choice for this project:
- **Performance**: Zero-cost abstractions, no GC pauses — critical for real-time agent loops
- **Safety**: Memory safety without runtime overhead — essential for concurrent sub-agent execution
- **Async**: tokio runtime provides robust async/concurrent primitives for streaming, parallel agents, and scheduling
- **Cross-platform**: UniFFI generates Kotlin (Android), Swift (iOS), C++/C# (desktop) bindings from a single Rust core
- **Type system**: Rust enums perfectly model the sealed-class pattern (ContentBlock, etc.); traits model the platform adaptation layer

### Core Traits (oneai-core)

```rust
/// LLM Provider — the primary abstraction for all model interactions
#[uniffi::export(with_foreign)]
pub trait LlmProvider: Send + Sync {
    async fn infer(&self, req: InferenceRequest) -> Result<InferenceResponse>;
    async fn infer_stream(&self, req: InferenceRequest) -> Result<InferenceStream>;  // Stream<SSEChunk>
    fn capabilities(&self) -> ModelCapability;
    fn config(&self) -> &ModelConfig;
}

/// Tool — unified interface for all tools (local, MCP, platform)
#[uniffi::export(with_foreign)]
pub trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn parameters_schema(&self) -> serde_json::Value;  // JSON Schema
    fn risk_level(&self) -> RiskLevel;  // Low / Medium / High
    async fn execute(&self, args: serde_json::Value) -> Result<ToolOutput>;
}

/// Memory Store — abstraction for both short-term and long-term
pub trait MemoryStore: Send + Sync {
    async fn store(&self, entry: MemoryEntry) -> Result<()>;
    async fn retrieve(&self, query: MemoryQuery, top_k: usize) -> Result<Vec<MemoryEntry>>;
    async fn compress(&self, threshold: usize) -> Result<Vec<MemoryEntry>>;
}

/// Skill Provider
pub trait SkillProvider: Send + Sync {
    async fn select_skills(&self, user_input: &str, top_k: usize) -> Result<Vec<SkillDescriptor>>;
    fn register_skill(&self, skill: SkillDescriptor) -> Result<()>;
    fn remove_skill(&self, name: &str) -> Result<()>;
}

/// Platform Tool — platform-specific tool interface, implemented per platform
pub trait PlatformTool: Tool {
    fn platform(&self) -> Platform;  // Windows / Mac / Linux / Android / Harmony / iOS
}
```

### ScopeState & Reducer (oneai-agent)

```rust
    fn reduce(&self, global: &mut GlobalState, reductions: Vec<Reduction>) -> Result<()>;
}
```

## Cross-Platform Binding Strategy (UniFFI)

Use UniFFI **proc-macro mode** (not UDL) for ergonomic Rust definitions:

1. **oneai-uniffi crate**: Contains all `#[uniffi::export]` annotations on traits, structs, enums. This is the single source for all bindings.
2. **Generation**: `cargo build` generates Kotlin, Swift, C++, C# headers automatically.
3. **Platform wrappers (platforms/)**: Each platform directory contains thin native wrappers that:
   - Convert UniFFI callbacks to platform-native async patterns (Kotlin Coroutines, Swift async/await, C# Task)
   - Implement `PlatformTool` for platform-specific functionality (PowerShell, AppleScript, WorkManager, etc.)
   - Implement `ApprovalGate` with native UI dialogs
   - Implement `TaskScheduler` with platform-native scheduling (WorkManager, daemon, etc.)

4. **HarmonyOS strategy**: Use `napi-ohos` for ArkTS bindings (UniFFI doesn't natively support HarmonyOS). The Rust core compiles via OpenHarmony NDK (`ohos-rs` cross-compilation target), and `napi-ohos` generates the ArkTS bridge.

## Key Design Decisions

- **LLM Provider Priority**: Phase 1 implements both **OpenAI-compatible protocol** (covers OpenAI, DeepSeek, 智谱, etc.) and **Anthropic Claude native protocol** (covers Claude models) simultaneously. Ollama local provider follows as Step 1.5.
- **Vector Storage**: Uses an **embedded lightweight HNSW implementation** (`instant-hnsw` or `rust-hnsw`) as the default long-term memory store — zero external dependency, suitable for mobile deployment. A `VectorStore` trait allows future extension to remote databases (Qdrant, Milvus) when needed.
- **HarmonyOS Binding**: UniFFI doesn't natively support HarmonyOS. Strategy: compile Rust core via OpenHarmony NDK (`ohos-rs`), then use `napi-ohos` to generate ArkTS bridge — a proven approach used by existing HarmonyOS Rust projects.
- **Async Pattern**: All core operations use `tokio` async. UniFFI exposes async methods that platform wrappers convert to native async patterns (Kotlin Coroutines, Swift async/await, C# Task).
- **Parser Strategy**: Layer 1 (constrained decoding) is a trait abstraction in Phase 1, with actual BNF grammar integration deferred to when a constrained-decoding provider is connected. Layers 2+3 are fully functional from Phase 1.

## Verification Strategy

Each phase has integration tests:

- **Phase 1**: Send multimodal inference request via cloud provider → receive streaming response → parse tool call correctly
- **Phase 2**: Submit complex task → verify Plan decomposition → verify parallel sub-agent execution → verify ScopeState isolation → verify Reflection result check
- **Phase 3**: Run agent loop with tool calls → verify sliding window memory → trigger high-risk tool → verify approval gate suspends → approve → verify execution resumes
- **Phase 4**: Define workflow in YAML → compile → validate (detect intentional dead loop) → execute workflow → verify skill injection in agent context
- **Phase 5**: Start agent loop → simulate interruption → verify checkpoint → recover → verify loop continues correctly
- **Phase 6**: Run Android example app → verify Kotlin Coroutine integration → verify native approval dialog → verify WorkManager scheduling

Additional continuous verification:
- `cargo test` across all crates on each commit
- `cargo clippy` for lint enforcement
- Cross-compilation CI: build for `x86_64-pc-windows-msvc`, `aarch64-linux-android`, `aarch64-apple-ios`, and HarmonyOS NDK target

## Summary

This plan builds OneAI in 6 phases, each independently useful:
1. **Foundation** — types, LLM provider, parsing (can talk to models)
2. **Agents** — paradigm fusion with state isolation (can run intelligent loops)
3. **Memory/Tools/Approval** — persistence, tooling, safety (can remember, act, and ask permission)
4. **Workflow/Skill/RAG** — structured workflows, dynamic skills, document retrieval (can plan and specialize)
5. **Scheduler/Persistence** — timing and recovery (can persist and resume)
6. **Bindings/Platform** — cross-platform deployment (can run everywhere)

The Rust core is fully platform-independent; platform-specific code lives in thin native wrappers that implement core traits (`PlatformTool`, `ApprovalGate`, `TaskScheduler`). UniFFI proc-macros generate all foreign bindings from the Rust definitions, and HarmonyOS uses napi-ohos as its bridge technology.
